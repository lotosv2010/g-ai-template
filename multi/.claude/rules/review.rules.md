# 审查规则

> AI 在完成任务后必须按照本清单自审，确认全部通过后才能提交。

## 自审清单

### 1. 正确性

- [ ] 代码实现是否完整满足任务需求
- [ ] 是否处理了所有边界情况
- [ ] 是否有遗漏的错误处理路径
- [ ] 逻辑是否正确，有无 off-by-one 等低级错误

### 2. 安全性

- [ ] 是否存在注入风险（SQL / XSS / 命令注入）
- [ ] 用户输入是否经过校验和消毒
- [ ] 是否泄露了敏感信息（日志、错误消息、注释中）
- [ ] 权限检查是否完整

### 3. 架构合规

- [ ] 是否遵守了分层依赖规则
- [ ] 是否保持了模块边界
- [ ] 是否在允许的文件范围内修改（参见 `file-scope.md`）
- [ ] 是否遵循了命名规范（参见 `naming.md`）
- [ ] 是否触碰了禁止项（参见 `forbidden.md`）

### 4. 代码质量

- [ ] 函数是否足够短小（不超过 50 行）
- [ ] 嵌套深度是否合理（不超过 3 层）
- [ ] 是否存在重复代码可以提取
- [ ] 命名是否清晰表达意图
- [ ] 是否有不必要的复杂度

### 5. 测试

- [ ] 是否为新增逻辑编写了测试
- [ ] 现有测试是否仍然通过
- [ ] 测试是否覆盖了核心路径和边界情况

### 6. 文档与记忆

- [ ] 涉及 API 变更时，`API_SPEC.md` 是否已更新
- [ ] 涉及模型变更时，`DATA_MODEL.md` 是否已更新
- [ ] 是否有值得记录的新模式或问题需要更新 `.claude/memory/`
- [ ] `docs/tasks/BOARD.md` 任务状态是否已更新
- [ ] `docs/tasks/CURRENT.md` 索引是否已更新

## 审查不通过的处理

如果任何一项检查未通过：

1. 修复问题
2. 重新执行自审清单
3. 在提交信息中说明修复内容
