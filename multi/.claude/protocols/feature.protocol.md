# 功能开发协议

> 新增功能时遵循此协议。在通用任务协议基础上增加以下步骤。

## 前置检查

1. **确认产品范围**: 检查 `docs/PRODUCT.md`，确认功能在"要做的"列表中
2. **确认架构适配**: 检查 `docs/ARCHITECTURE.md`，确认功能在现有架构下可实现
3. **确认 API 契约**: 如涉及接口，先设计 API 并更新 `docs/API_SPEC.md`
4. **确认数据模型**: 如涉及数据变更，先更新 `docs/DATA_MODEL.md`

## 开发顺序

遵循由内向外的开发顺序:

```
1. Domain 层 —— 定义实体、值对象、领域事件
2. Service 层 —— 实现业务逻辑
3. Repository 层 —— 实现数据访问
4. Controller 层 —— 实现 HTTP 入口
5. 测试 —— 编写单元测试和集成测试
```

## 每一层的检查点

### Domain 层

- [ ] 实体和值对象是否有完整的校验逻辑
- [ ] 领域规则是否在领域层内闭合
- [ ] 是否依赖了任何外部框架（不应该）

### Service 层

- [ ] 业务逻辑是否完整
- [ ] 是否正确使用了 Repository 接口（而非具体实现）
- [ ] 错误情况是否都有处理

### Repository 层

- [ ] 查询是否高效（有无 N+1 问题）
- [ ] 是否使用了参数化查询
- [ ] 事务边界是否正确

### Controller 层

- [ ] 请求参数是否有校验
- [ ] 响应格式是否符合 `API_SPEC.md`
- [ ] 权限检查是否到位

### 测试

- [ ] 核心业务逻辑的单元测试
- [ ] 接口的集成测试
- [ ] 边界情况的测试
