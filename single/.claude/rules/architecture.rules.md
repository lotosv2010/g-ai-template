# 架构规则

> 硬性规则。AI 在涉及架构层面的改动时必须遵守，无例外。

## 分层规则

1. **依赖方向**: 只允许外层依赖内层，禁止反向依赖
   - `controller -> service -> domain` (允许)
   - `domain -> service` (禁止)
2. **领域纯净**: `domain/` 层禁止引入任何框架代码或基础设施依赖
3. **接口隔离**: 层与层之间通过接口通信，禁止直接引用具体实现

## 模块规则

1. **模块自治**: 每个功能模块只能访问自己的数据，禁止跨模块直接查库
2. **模块通信**: 模块间通过公开的 Service 接口或事件通信，禁止直接导入内部实现
3. **循环依赖**: 模块间禁止循环依赖，如需互相调用则引入事件或中间层

## 变更规则

1. **架构变更需 ADR**: 任何架构级变更（新增层、改变依赖方向、引入新模式）必须先创建 ADR
2. **技术栈变更需审批**: 引入新框架或替换现有技术必须经过人工审批
3. **数据库变更需先更新文档**: 修改 Schema 前必须先更新 `docs/DATA_MODEL.md`
4. **API 变更需先更新契约**: 修改接口前必须先更新 `docs/API_SPEC.md`

## 禁止项

1. **禁止全局状态**: 不使用全局变量或单例模式传递可变状态
2. **禁止 God Class/Function**: 单个类或函数承担过多职责时必须拆分
3. **禁止绕过层级**: 例如 Controller 直接访问 Repository
4. **禁止隐式耦合**: 不通过文件路径约定、魔法字符串等方式建立耦合关系
